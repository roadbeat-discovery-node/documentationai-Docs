---
title: Faceted Filtering
description: Aggregation-based filter facets for content discovery UIs.
---

Facets are aggregation-powered filter options returned alongside search results. They enable filter sidebars in client applications — showing categories, topics, publishers, date ranges, and more with accurate document counts.

## How facets work

When a search request is executed, the Discovery Node automatically includes Elasticsearch aggregations based on the searched content types. The raw aggregation results are transformed into a clean, labeled format suitable for UI rendering.

```mermaid
graph LR
    A[Search Request] --> B[Build ES Aggregations]
    B --> C[Execute Search]
    C --> D[Transform Aggregations]
    D --> E[Labeled Facet Response]
```

## Global facets

These 11 facets are available for all content types:

| Facet | Field | Type | Default | Buckets |
|-------|-------|------|---------|---------|
| **Categories** | `classification.categories` | terms | Yes | 20 |
| **Topics** | `classification.topics` | terms | Yes | 50 |
| **Publishers** | `publisher.name` | terms | Yes | 20 |
| **Languages** | `access.languages` | terms | Yes | 30 |
| **Content Types** | `content_type` | terms | Yes | 20 |
| **Publication Date** | `temporal.date_published` | date_histogram | Yes | monthly |
| **Countries** | `geographic.country_codes` | terms | No | 50 |
| **Access Type** | `access.access_type` | terms | No | 10 |
| **Free Content** | `access.is_free` | terms | No | 2 |
| **Publisher Type** | `publisher.type` | terms | No | 10 |
| **Trust Level** | `publisher.trust_level` | terms | No | 5 |

<Callout kind="info">
  Facets marked as "Default: Yes" are automatically included in search responses. Non-default facets can be requested via custom aggregation definitions in the search request.
</Callout>

## Content-type-specific facets (schema-driven)

When searching a **single content type**, the Discovery Node dynamically adds type-specific facets derived from `x-filter` annotations in the content type's metadata schema. No hard-coded per-type configuration is needed — the `DynamicFacetService` reads the schema from the Schema Registry, walks its properties, and builds facet definitions automatically.

```mermaid
graph LR
    A[Single content type query] --> B[DynamicFacetService]
    B --> C[Fetch metadata schema from Registry]
    C --> D[Walk properties for x-filter]
    D --> E[Build FacetDefinitions]
    E --> F[Add to ES aggregations]
```

Type-specific facet values are stored in the `type_filters` sub-object of each teaser document. This sub-object uses `dynamic: true` in the ES mapping, so new filter fields are indexed automatically without template changes.

<Callout kind="tip">
  To add new filters to any content type, simply add an `x-filter` annotation to the relevant property in its metadata schema. No code changes required in the Discovery Node, Studio, or frontend.
</Callout>

### Example: Event type-specific facets

When filtering to `content_types: ["event"]`, the following facets are automatically added based on the event metadata schema's `x-filter` annotations:

| Facet | ES Field | Type | UI Hint |
|-------|----------|------|---------|
| **Event Type** | `type_filters.event_type` | terms | checkboxes |
| **Format** | `type_filters.event_format` | terms | chips |
| **Category** | `type_filters.event_category` | terms | searchable_checkboxes |
| **Event Date** | `type_filters.event_date` | date_histogram | date_picker |
| **Free Event** | `type_filters.is_free` | terms (boolean) | toggle |
| **Audience** | `type_filters.audience_type` | terms | checkboxes |
| **Skill Level** | `type_filters.skill_level` | terms | chips |
| **Status** | `type_filters.event_status` | terms | chips |

### Type-specific facet metadata

Type-specific facets include additional metadata fields in the response to help frontends render appropriate widgets:

| Field | Type | Description |
|-------|------|-------------|
| `content_type_specific` | boolean | `true` for schema-driven facets |
| `ui_hint` | string | Widget hint: `chips`, `checkboxes`, `searchable_checkboxes`, `toggle`, `slider`, `date_picker`, `star_rating` |
| `unit` | string | Unit label for range facets (e.g., `"m²"`, `"min"`, `"€"`) |
| `labels` | object | Multilingual labels from the schema (e.g., `{ "en": "Event Type", "de": "Veranstaltungstyp" }`) |
| `priority` | number | Display priority — lower values appear higher in the sidebar |

## Facet response format

Facets are returned in the `aggregations` array of every search response:

```json
{
  "aggregations": [
    {
      "field": "classification.categories",
      "label": "Categories",
      "type": "terms",
      "buckets": [
        { "key": "politics", "doc_count": 456, "label": "politics" },
        { "key": "environment", "doc_count": 389, "label": "environment" },
        { "key": "technology", "doc_count": 234, "label": "technology" }
      ],
      "sum_other_doc_count": 155
    },
    {
      "field": "temporal.date_published",
      "label": "Publication Date",
      "type": "date_histogram",
      "buckets": [
        { "key": "2026-01", "doc_count": 1234 },
        { "key": "2026-02", "doc_count": 987 }
      ]
    },
    {
      "field": "geographic.country_codes",
      "label": "Countries",
      "type": "terms",
      "buckets": [
        { "key": "DE", "doc_count": 3456, "label": "Germany" },
        { "key": "AT", "doc_count": 1234, "label": "Austria" },
        { "key": "CH", "doc_count": 890, "label": "Switzerland" }
      ]
    }
  ]
}
```

### Facet types

| Type | Description | Example |
|------|-------------|---------|
| **terms** | Discrete value buckets with doc counts | Categories, publishers, languages |
| **date_histogram** | Time-based buckets (day, week, month) | Publication date, event date |
| **geo_distance** | Distance range buckets from a center point | "Within 10km", "Within 50km" |
| **range** | Numeric range buckets | Price ranges, word count |

## Custom aggregations

You can override the default facets by providing custom aggregation definitions in the search request:

```json
{
  "query": { "text": "restaurants Berlin" },
  "content_types": ["restaurants"],
  "aggregations": {
    "cuisine": {
      "type": "terms",
      "field": "classification.cuisine_type",
      "size": 15
    },
    "price_range": {
      "type": "range",
      "field": "pricing.average_price"
    }
  }
}
```

## Country label mapping

Country code facet buckets are automatically mapped to human-readable labels for all EU/EEA countries. For example, `DE` becomes `Germany`, `FR` becomes `France`, `AT` becomes `Austria`.

Boolean facets (`true`/`false` keys) are labeled as `Yes`/`No`.

## Boolean filter handling

Boolean fields in `type_filters` (e.g., `is_breaking`, `is_exclusive`, `is_free`) are stored as native ES boolean values (`true`/`false`). However, frontends typically send filter values as strings (`"1"`, `"0"`).

The Discovery Node's query builder automatically coerces these string values to proper booleans before building Elasticsearch queries:

| Frontend value | ES query value |
|---------------|---------------|
| `"1"` | `true` |
| `"true"` | `true` |
| `"0"` | `false` |
| `"false"` | `false` |

This coercion is applied transparently in `applyTypeFilters()` — frontends do not need to handle this conversion.

<Callout kind="info">
  Non-boolean string values (e.g., `"analysis"`, `"free"`) pass through the coercion unchanged and are matched as keywords.
</Callout>
